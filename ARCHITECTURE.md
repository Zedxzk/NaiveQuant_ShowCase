# NaiveQuant 架构说明

这份文档介绍 NaiveQuant 当前的分层架构和几个关键边界。本文档从技术评审角度出发，仅说明公开结构，不展开专有实现细节。

## 1. 分层依赖模型

平台依赖自下而上组织：底层提供基础能力，上层组合业务流程。在最近一次依赖图检查中，没有发现跨层反向依赖，也未发现跨模块循环导入。

```
┌─────────────────────────────────────────────────────────────────────┐
│ L5 交互层   ui/ · cli_commands/ · reporting/ · visualization/          │
├─────────────────────────────────────────────────────────────────────┤
│ L4 编排层   ml/ · backtest/ · workflows/ · selection/ · optimization/  │
├─────────────────────────────────────────────────────────────────────┤
│ L3 领域层   factor/  ‖  strategy/  ‖  trading/(execution·account·risk) │
│             └──────── 三族平行、互不导入 ────────┘                    │
├─────────────────────────────────────────────────────────────────────┤
│ L2 数据/模型 data/ · models/(rl·deep·determinism) · stats · auth        │
├─────────────────────────────────────────────────────────────────────┤
│ L1 core 基础 schemas · secrets · settings_store · workspace ·          │
│              data_catalog · protocols                                  │
├─────────────────────────────────────────────────────────────────────┤
│ L0 零依赖层  constants(零依赖·无内部引用) · logger · config            │
└─────────────────────────────────────────────────────────────────────┘
```

被依赖最多的模块是 `core/workspace`、`core/constants`、`trading`、`auth`、`core/logger` 和 `core/config`。这些模块位于基础层，改动影响范围较大，因此接口设计力求稳定。

## 2. `quantplatform` 和 `trading_server` 的边界

运维/推理服务 `trading_server` 是一个独立包。它可以调用研究平台 `quantplatform` 的公开接口，但 `quantplatform` 不反向导入 `trading_server`：

```
trading_server  ──单向──►  quantplatform.*
```

服务端主要依赖几类稳定接口：records、broker/env、auth 和 core 常量。早期曾有一些直接调用内部实现的代码，后来统一收敛到公开 facade 函数，避免 `quantplatform` 内部重构时波及服务端。

## 3. YAML 驱动的数据流水线

```
   config/*.yaml
        │
        ▼
  ┌───────────┐   统一协议    ┌───────────┐   统一协议   ┌───────────┐
  │  数据层    │ ───────────► │ Alpha/ML  │ ──────────► │  组合构建  │
  └───────────┘              └───────────┘             └─────┬─────┘
                                                             ▼
  ┌───────────┐   账户决策    ┌───────────┐   目标权重   ┌───────────┐
  │  执行下单  │ ◄─────────── │   风控     │ ◄────────── │   策略     │
  └───────────┘              └───────────┘             └───────────┘
        │
        ▼
   券商适配器(Alpaca/Futu/IBKR/Longbridge/QMT) → 报告 → 运维监控
```

执行层仅接收最终账户决策，不包含具体交易规则。研究阶段产出的目标权重和风控后的账户决策，可直接复用于纸账户或实盘路径，无需重写下单逻辑。

## 4. DuckDB 存储和并发

DuckDB 的写入模型更适合单进程分析场景，但长期运行的采集服务与研究进程会并发访问数据。具体处理方式如下：

- **写入器不长期持有锁**：`open → write → close`，每次写完即释放连接。
- **只读连接复用**：市场数据这类静态库在进程内复用只读句柄，降低磁盘 I/O 较慢时反复建连的开销。
- **记录层采用 append-only JSONL**：订单、对账和审计记录避免与 DuckDB 竞争写锁，跨进程追加文件即可。

## 5. Point-in-Time 数据处理

量化研究中最容易忽略的问题之一是前视偏差。平台在采集层就已标记数据“属于哪个财报周期”和“何时对外可见”：

- SEC EDGAR 财报同时保存 `period_end` 和 `filed`→`available_at`。
- 因子和回测行至任一历史时点，只能读取当时已经公布的数据。
- 这一约束在数据层就已落地，后续的因子、回测和组合模块均按同一套时间口径取数。

## 6. 部署拓扑

```
        本地开发机 (main 分支)
            │  git push + rsync 部署链
            ├──────────────────────────┐
            ▼                          ▼
   ┌─────────────────┐        ┌─────────────────┐
   │  云服务器 A       │        │  云服务器 B       │
   │  systemd --user  │        │  systemd --user  │
   │  ├ nquant-ui     │        │  ├ nquant-ui     │
   │  ├ trading-server│        │  ├ trading-server│
   │  ├ recorder ×2   │        │  ├ recorder ×2   │
   │  └ enrich   ×2   │        │  └ enrich   ×2   │
   └─────────────────┘        └─────────────────┘
     服务绑定 127.0.0.1，经 SSH 隧道访问，不暴露公网
```

- systemd `--user` + linger：免登录常驻，`Restart=on-failure` 自动重启。
- 凭证主口令仅保存在 systemd 用户管理器内存中，通过 `set-environment` 传递给服务，不写入磁盘。
- 部署使用增量 rsync，排除 `.git`、数据和快照目录，安装后进行冒烟测试和版本核对。
