# NaiveQuant 架构说明

这份文档记录 NaiveQuant 当前的分层方式和几个关键边界。它偏向技术评审视角，只讲公开结构，不展开专有实现细节。

## 1. 分层依赖模型

平台按自下而上的方向组织依赖：底层提供基础能力，上层组合业务流程。最近一次依赖图检查里，没有发现跨层反向依赖，也没有顶层 import 环。

```
┌─────────────────────────────────────────────────────────────────────┐
│ L5 交互层   ui/ · cli_commands/ · reporting/ · visualization/          │
├─────────────────────────────────────────────────────────────────────┤
│ L4 编排层   ml/ · backtest/ · workflows/ · selection/ · optimization/  │
├─────────────────────────────────────────────────────────────────────┤
│ L3 领域层   factor/  ‖  strategy/  ‖  trading/(execution·account·risk) │
│             └──────── 三族平行、互不 import ────────┘                    │
├─────────────────────────────────────────────────────────────────────┤
│ L2 数据/模型 data/ · models/(rl·deep·determinism) · stats · auth        │
├─────────────────────────────────────────────────────────────────────┤
│ L1 core 基础 schemas · secrets · settings_store · workspace ·          │
│              data_catalog · protocols                                  │
├─────────────────────────────────────────────────────────────────────┤
│ L0 叶子     constants(纯叶子·0 内部 import) · logger · config           │
└─────────────────────────────────────────────────────────────────────┘
```

被依赖最多的模块是 `core/workspace`、`core/constants`、`trading`、`auth`、`core/logger` 和 `core/config`。这些模块属于基础层代码，改动影响范围较大，因此接口会尽量保持稳定。

## 2. `quantplatform` 和 `trading_server` 的边界

运维/推断服务 `trading_server` 是一个独立包。它可以调用研究平台 `quantplatform` 的公开入口，但 `quantplatform` 不反向导入 `trading_server`：

```
trading_server  ──单向──►  quantplatform.*
```

服务端主要依赖几类稳定入口：records、broker/env、auth 和 core 常量。早期存在一些直接调用内部实现的代码，后来统一收敛为公开门面函数，避免 `quantplatform` 内部重构时波及服务端。

## 3. YAML 驱动的数据流水线

```
   config/*.yaml
        │
        ▼
  ┌───────────┐   统一协议    ┌───────────┐   统一协议   ┌───────────┐
  │  数据层    │ ───────────► │ Alpha/ML  │ ──────────► │  组合构建  │
  └───────────┘              └───────────┘             └─────┬─────┘
                                                             ▼
  ┌───────────┐   账户决策    ┌───────────┐   目标权重   ┌───────────┐
  │  执行下单  │ ◄─────────── │   风控     │ ◄────────── │   策略     │
  └───────────┘              └───────────┘             └───────────┘
        │
        ▼
   券商适配器(Alpaca/Futu/IBKR/Longbridge/QMT) → 报告 → 运维监控
```

执行层只消费最终账户决策，不内置具体交易规则。策略在研究阶段产出的目标权重和风控后的账户决策，复用到纸账户或实盘路径时不需要重新写一套下单逻辑。

## 4. DuckDB 存储和并发

DuckDB 的写入模型更适合单进程分析场景，但长期运行的采集服务与研究进程会并发访问数据。这里的处理方式是：

- **写入器不长期占锁**：`open → write → close`，每次写完即释放连接。
- **只读连接复用**：市场数据这类静态库在进程内复用只读句柄，降低慢速磁盘上的重复连接开销。
- **记录层走 append-only JSONL**：订单、对账和审计记录避免与 DuckDB 竞争写锁，跨进程追加文件即可。

## 5. Point-in-Time 数据处理

量化研究中最容易被忽视的问题之一是前视偏差。平台在采集层就记录数据“属于哪个财报周期”和“何时对外可见”：

- SEC EDGAR 财报同时保存 `period_end` 和 `filed`→`available_at`。
- 因子和回测在某个历史时点，只能读取当时已经公布的数据。
- 这一约束落实在数据层，后续的因子、回测和组合模块均按同一套时间口径取数。

## 6. 部署拓扑

```
        本地开发机 (main 分支)
            │  git push + rsync 部署链
            ├──────────────────────────┐
            ▼                          ▼
   ┌─────────────────┐        ┌─────────────────┐
   │  云服务器 A       │        │  云服务器 B       │
   │  systemd --user  │        │  systemd --user  │
   │  ├ nquant-ui     │        │  ├ nquant-ui     │
   │  ├ trading-server│        │  ├ trading-server│
   │  ├ recorder ×2   │        │  ├ recorder ×2   │
   │  └ enrich   ×2   │        │  └ enrich   ×2   │
   └─────────────────┘        └─────────────────┘
     服务绑定 127.0.0.1，经 SSH 隧道访问，不暴露公网
```

- systemd `--user` + linger：免登录常驻，`Restart=on-failure` 自动重启。
- 凭证主口令只放在 systemd 用户管理器内存里，通过 `set-environment` 传给服务，不写入磁盘。
- 部署采用增量 rsync，排除 `.git`、数据和快照目录，并在安装后做冒烟自检和版本核对。
