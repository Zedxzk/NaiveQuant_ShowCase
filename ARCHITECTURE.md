# 架构说明 · NaiveQuant

> 面向技术评审的架构概览。内容基于平台真实的依赖图审查结论，不含专有实现细节。

## 1. 分层依赖模型

平台遵循**自下而上的正向依赖**：底层薄、上层胖，未出现跨层反向违规，也无顶层 import 环。

```
┌─────────────────────────────────────────────────────────────────────┐
│ L5 交互层   ui/ · cli_commands/ · reporting/ · visualization/          │
├─────────────────────────────────────────────────────────────────────┤
│ L4 编排层   ml/ · backtest/ · workflows/ · selection/ · optimization/  │
├─────────────────────────────────────────────────────────────────────┤
│ L3 领域层   factor/  ‖  strategy/  ‖  trading/(execution·account·risk) │
│             └──────── 三族平行、互不 import ────────┘                    │
├─────────────────────────────────────────────────────────────────────┤
│ L2 数据/模型 data/ · models/(rl·deep·determinism) · stats · auth        │
├─────────────────────────────────────────────────────────────────────┤
│ L1 core 基础 schemas · secrets · settings_store · workspace ·          │
│              data_catalog · protocols                                  │
├─────────────────────────────────────────────────────────────────────┤
│ L0 叶子     constants(纯叶子·0 内部 import) · logger · config           │
└─────────────────────────────────────────────────────────────────────┘
```

**枢纽模块（扇入最高）**：`core/workspace`、`core/constants`、`trading`、`auth`、`core/logger`、
`core/config`。这些是稳定的地基型模块 —— 改动面广、但本身极少改，符合健康的依赖形态。

## 2. 跨包边界：`quantplatform` ↔ `trading_server`

运维/推断服务 `trading_server` 是一个**独立包**，与研究平台 `quantplatform` 之间保持**严格单向依赖**：

```
trading_server  ──单向──►  quantplatform.*
   （quantplatform 完全不反向 import trading_server）
```

且依赖收敛在少数**稳定门面**上（records / broker&env / auth / core 常量）。任何 `quantplatform`
内部实现的重构，都不应打穿这条边界 —— 为此专门把服务端对内部私有实现的深钻依赖，**收口为公开门面函数**。

## 3. 数据流水线（YAML 驱动）

```
   config/*.yaml
        │
        ▼
  ┌───────────┐   统一协议    ┌───────────┐   统一协议   ┌───────────┐
  │  数据层    │ ───────────► │ Alpha/ML  │ ──────────► │  组合构建  │
  └───────────┘              └───────────┘             └─────┬─────┘
                                                             ▼
  ┌───────────┐   账户决策    ┌───────────┐   目标权重   ┌───────────┐
  │  执行下单  │ ◄─────────── │   风控     │ ◄────────── │   策略     │
  └───────────┘              └───────────┘             └───────────┘
        │
        ▼
   券商适配器(Alpaca/Futu/IBKR/Longbridge/QMT) → 报告 → 运维监控
```

**关键设计**：执行层只消费「最终账户决策」，不内置具体交易规则。研究阶段产出的策略，
可零改动切换到实盘执行 —— 这是训练/推断「零 skew」的基础。

## 4. 存储与并发：DuckDB「开→写→关」

DuckDB 是单写者模型。平台的数据采集守护是长期写入方，而研究 / Agent 侧需要随时并发只读。
解决方案：

- **写入器不驻留写锁**：`open → write → close` 短事务，写完即释放。
- **只读复用连接**：静态只读库（如市场数据）进程级复用同一句柄，避免慢盘上每次 ~秒级的重连开销。
- **记录层 append-only JSONL**：订单/对账/审计等跨进程记录走 append-only 文件，天然无写锁竞争，
  彻底绕开 DuckDB 单写者限制。

## 5. 数据正确性：Point-in-Time

量化研究最隐蔽的坑是**前视偏差**。平台在数据层就把它锁死：

- SEC EDGAR 财报同时记录 `period_end`（财报周期）与 `filed`→`available_at`（对外公布日）。
- 因子/回测在任意历史时点，只能看到「该时点**已公布**」的数据，而非「该周期最终**修订后**」的数据。
- 这套机制在采集阶段就固化，下游所有研究天然 PIT 正确。

## 6. 部署拓扑

```
        本地开发机 (main 分支)
            │  git push + rsync 部署链
            ├──────────────────────────┐
            ▼                          ▼
   ┌─────────────────┐        ┌─────────────────┐
   │  云服务器 A       │        │  云服务器 B       │
   │  systemd --user  │        │  systemd --user  │
   │  ├ nquant-ui     │        │  ├ nquant-ui     │
   │  ├ trading-server│        │  ├ trading-server│
   │  ├ recorder ×2   │        │  ├ recorder ×2   │
   │  └ enrich   ×2   │        │  └ enrich   ×2   │
   └─────────────────┘        └─────────────────┘
     服务全绑 127.0.0.1，经 SSH 隧道访问，不暴露公网
```

- systemd `--user` + linger：免登录常驻、`Restart=on-failure` 崩溃自愈。
- 凭证主口令仅存于 systemd 用户管理器**内存**（`set-environment`），跨服务重启继承、不落盘。
- 部署为增量 rsync（排除 .git / 数据 / 快照），配套安装冒烟自检与版本核对。
